# BotBand
## 🤖 Telegram Multi-Agent Discussion System
Проєкт являє собою інтелектуальну систему автоматизації дискусій у Telegram-групах, побудовану на базі декількох сесій користувачів (UserBots). Система імітує реальне спілкування між декількома учасниками, використовуючи сучасні LLM (Large Language Models) для генерації контекстуальних відповідей.
Основна мета системи — підтримання активності у спільнотах, створення змістовних дискусій на задані теми та автоматизація взаємодії з користувачами. Завдяки використанню декількох акаунтів одночасно, боти можуть взаємодіяти не лише з реальними користувачами, а й між собою, створюючи ефект "живого" чату.

## 🏗 Архітектура системи
Проєкт реалізовано на базі Event-Driven Multi-Agent Architecture (подійно-орієнтованої мультиагентної архітектури).

### Ключові компоненти:
- Telegram Manager: Центральний вузол, що керує життєвим циклом декількох клієнтів Pyrogram.
- LLM Service: Модуль взаємодії з нейромережами для формування відповідей з урахуванням ролі кожного бота.
- Context Manager (Redis): Використовується як спільне сховище стану (Shared State). Всі боти мають доступ до єдиної історії повідомлень та поточної теми обговорення.
- Decentralized Logic: Кожен бот самостійно приймає рішення про відповідь на основі ймовірнісних алгоритмів та аналізу черговості (щоб уникнути монологів одного акаунта).

### Переваги архітектури:
1. Масштабованість: Можливість запуску необмеженої кількості ботів без зміни логіки ядра.
2. Стійкість до блокувань: Розподіл активності між декількома акаунтами знижує ризик отримання Flood Wait від Telegram.
3. Низька зв'язність: Компоненти системи незалежні, що дозволяє легко замінювати базу даних або модель LLM.
4. Синхронізація через події: Використання асинхронності (asyncio) забезпечує високу швидкість реакції навіть при великій кількості вхідних повідомлень.

## 🛠 Порядок запуску
Для коректної роботи системи необхідно виконати налаштування оточення та підготовку сесій.

### 1. Підготовка сесій акаунтів
Система працює з готовими файлами сесій Pyrogram (.session).
- Якщо у вас вже є сесії: Помістіть їх у директорію, вказану в змінній TELEGRAM_WORKDIR (за замовчуванням ./sessions/).
- Якщо сесій немає:
    1. Налаштуйте файл .env_auth (використовуйте .env_auth.example як зразок), вказавши API_ID та API_HASH.
    2. Запустіть скрипт авторизації: python auth.py.
    3. Дотримуйтесь інструкцій у терміналі для створення нових файлів сесій.

### 2. Налаштування основних параметрів
Створіть файл .env у корені проєкту (на основі .env.example) та заповніть наступні блоки:

- REDIS: параметри підключення до сховища.
- TELEGRAM: загальні налаштування API.
- GROUP: ID цільової групи та ID адміністратора.
- LLM: параметри підключення до API нейромережі (Base URL, Key, Model).
- BEHAVIOR: налаштування затримок та перелік назв сесій ботів.

### 3. Запуск через Docker Compose
Проєкт повністю контейнеризований для забезпечення стабільності середовища.

```Bash
docker compose up --build -d
```
Команда автоматично запустить Redis та основний додаток з ботами.

## ⚙️ Логіка взаємодії
У системі реалізовано механізм динамічної зміни теми обговорення через вхідні події:

- Роль Адміністратора: Один із запущених акаунтів (чиє ID вказано як ADMIN_ID) виступає в ролі модератора тем.
- Механізм активації: Коли будь-який користувач Telegram надсилає приватне повідомлення (Direct Message) на цей адмін-акаунт, система автоматично:
    1. Зчитує текст повідомлення як нову "Глобальну тему".
    2. Оновлює стан у базі даних (Redis).
    3. Публікує офіційне оголошення в групі від імені адмін-акаунта про зміну вектору дискусії.
- Результат: Всі інші боти миттєво отримують оновлений контекст і починають генерувати відповіді, спираючись на нову тему, запропоновану стороннім користувачем.
